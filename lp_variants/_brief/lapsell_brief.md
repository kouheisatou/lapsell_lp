## 0. このLPの目的（販売者向け）
無名〜中堅の**ライブアイドル／インディーズ楽曲アーティスト・バンド／インディーズお笑い芸人**本人や小規模事務所に「練習の時間を出品してみよう」と思わせ、無料登録/問い合わせへ導く。流れ：①SEOで見つかる→②課題に共感→③仕組みを理解→④「本当に売れる？」を根拠でつぶす→⑤不安（事故・手間・費用）を消す→⑥行動。

## 1. Lapsell とは（サービスの核）
**「練習の“時間”」を売る、新しい形のランダムグッズ市場。** 加工なしの“日常・裏側”を、ガチャ型のデジタルグッズとして売買する。
- 流れ：**時間枠を出品 → ファンが（中身を知らずに）購入 → 当日スマホで撮るだけ → 撮れた映像が、開けるまで中身の分からない“ランダムグッズ”として届く。**
- 3つの肝：①**時間を“先に”売る**（完成品でなく未来の練習時間）／②**中身は運しだい＝ガチャ**（何が映るかは届くまで不明。構造そのものがガチャ）／③**届くのは世界に1つ**（落札者だけが持つ一点もののデジタルグッズ）。
- 一言：**「練習の時間を、収益に変える」**

## 2. 課題（2つの側面 — どちらもLPで描く）
- **課題①｜ファン**：“素の日常”に手が届かない。｛完成品しか届かない／現場（特典会・ライブ）でしか会えない＝地理と時間に縛られる／応援した証が残らない（投げ銭も配信も流れて消える）｝。**「近さ」と「自分だけの所有」を同時に満たす手段が存在しない。**
- **課題②｜アイドル・配信者（＝LPの読み手）**：一番価値ある“裏側”が収益にならない。｛撮影・編集・投稿が大変で続かない／ライブ配信は事故が怖い（失言・映り込みがそのまま流れる）／裏側は再生数が取れず儲からない＝だから撮られない｝。**毎日生まれる価値ある時間が、収益化されないまま捨てられている。**

## 3. 利用フロー（アプリ画面で必ず可視化＝HTMLモックで作る／§9参照）
**出品者側 4ステップ**：①出品（練習の日時と価格を決めて出品）→②購入（ファンが時間枠を購入・中身は不明）→③撮影（当日スマホで撮るだけ・編集不要）→④開封（後日ガチャのように開けて受け取る）。
**購入者の開封体験**：届く（あなたの区間が到着）→開ける（タップして開封）→受け取る（練習に思いがけず写る面白い瞬間。世界に一つだけ、あなたが所有）→SNS拡散（面白い瞬間がSNSへ＝新規ファンの入口）。
画面内テキストの基準値：出品=「6/20(金) 19:00-21:00」「¥300 / 1回」「542 / 720 区間」「出品する」／届く・開封=「あなたの区間が届きました」「タップして開ける」「開ける」（※「さあ、何かな…?」のような** AI臭い決め台詞は使わない**）／受け取り=「19:47:30 - 19:47:40 ダンス練習」「世界で一つだけ、あなたが所有」「再生」「シェア」。

## 4. 導入する4つの理由（=アピール①〜④。各セクションで根拠＋イメージ画像）
- **①「お金にならなかった時間」が売れる。** 捨てていた練習・裏側が新しい収益源に。根拠＝“素”には既に需要がある（§7の実績）＋ファンの約3人に1人が「払ってもいい」。
- **②下積み時代だからこそ、将来の価値上昇を見込んで買われる。** 「自分が育てた／見つけて支えた」という**物語ごと**ファンが買う。無名の今の一点ものが、ブレイクとともに価値を増す。（例：M!LKが7年下積みだった頃の練習雑談を“あなただけ”が持っていたら）
- **③自然に拡散され、宣伝になる。** ファンは自分だけが知る面白い瞬間を拡散したくなる。漏れて広まるのは損でなく“無料の宣伝”＝次のファンを呼ぶ。
- **④配信事故を事前に防げる。** ライブ配信と違い**撮影後にアップロード**。公開前に内容を確認でき、まずい瞬間は世に出ない（本当の事故の瞬間だけ非公開＋返金、偶然の名場面は届く）。

## 10. デザイン原則（全LP厳守）
- **線アイコンのみ**（inline SVG, stroke 1.5–2px, currentColor）。絵文字禁止。
- **「枠で囲みすぎる」AI臭を排除**：余白・タイポ階層・整列・背景帯で構造を作る。罫線/カードは参考サイトが実際に使う所だけ。影/角丸/グラデ枠の乱用禁止。
- **画像・図・アプリ画面を主役に**。抽象主張は必ず具体ビジュアルで。
- **【最重要】数字は「ドンと大きく出して後で説明」を禁止（AI臭の典型）。** 数字は必ず**わかる一文の中**に置くか、**グラフ＋説明見出し**で見せ、ぱっと見で「何の数字か・なぜ重要か」が伝わるように。実績も「20億回」を大きく出すだけにせず、結論を見出しに（例「『練習動画』が公式MVより観られている。」）し数字は文中で補強。
- **画像の比率・サイズ**：横長スクショ（YouTube実績≒4.5:1）は `width:100%;height:auto` で全体表示（cover切り抜き禁止＝見出し/数字が欠ける）。縦長と横長を同じグリッドに混ぜない（縦長は専用ハイライト枠）。どの画像も意図比率を保持（歪み・極端トリミング禁止）。
- **1ファイル完結**：CSS/JSはHTML埋め込み。**画像だけ外部**（自分の `assets/`）。生成画像は各LP専用、他LPと使い回さない。
- レスポンシブ必須（横スクロール禁止）。desktop 1440 / mobile 390 で破綻しないこと。

## 11. codex 画像生成の作法（①ヒーロー ②アピール①〜④イラスト ③アプリモック内に埋め込む画像。※アプリ画面“全体”はHTMLで作り、写真/イラスト部分だけcodex）
`~/.claude/skills/codex-imagegen/SKILL.md` 準拠。
```
codex exec --sandbox workspace-write --skip-git-repo-check '$imagegen <本文＋保存先絶対パス>' < /dev/null
```
- 冒頭 `$imagegen`。アスペクト比・解像度を本文に明記。生成手段はgpt-image直接出力のみ（SVG中間/スクリプト描画は禁止）。保存先絶対パスを明記。各LPの実測カラーを反映。
- **重要：codexはプロジェクトのルート（`.../lapsell_lp`）をcwdにして実行**（サブフォルダ実行で `.git` が作られ二重管理になる）。git関連コマンドは一切実行しない。git管理は親リポジトリ1つだけ。
- **【効率化・最重要】画像は「1回の `codex exec` で複数枚まとめて生成」する。** `codex exec` は毎回セッション起動コスト（数十秒）が大きく、1枚ごとに呼ぶと待ち時間が激増する。1プロンプトで複数画像を指示し、codexに1セッション内で `image_gen` を連続実行・各保存パスへ保存させる。例：`$imagegen 次の4枚を順に生成し各絶対パスに保存してください。1) <仕様> → <abs1> ／ 2) <仕様> → <abs2> ／ 3) … ／ 4) …`。目安：1LPの全画像を 3〜4回のcodex exec（アピール4枚で1回／課題2枚＋mock2枚で1回／対象3枚で1回／comparison・pricing・authenticityで1回）に束ねる。生成後にReadで確認し、崩れた画像だけ個別に再生成。
- 生成後はReadで確認。アピールイラストは概念が伝わればよい（画面内文字は最小限）。
